# Wix MCP Servers Exposed on CNA Websites

**Date:** 2026-04-24
**Context:** Well-known file scan of 487 CVE Numbering Authority (CNA) domains
**Finding:** 4 CNA websites expose unauthenticated MCP (Model Context Protocol) servers via the Wix platform

## Summary

While scanning CNA websites for well-known files (llms.txt, security.txt, robots.txt, etc.), we tested each domain for live MCP endpoints. **No CNA organization has intentionally deployed an MCP server on their main website.** However, 4 CNAs that host their sites on the Wix platform have MCP endpoints exposed automatically by Wix's platform-level feature.

## Affected Sites

| Domain | MCP Endpoint | Server Identity |
|--------|-------------|-----------------|
| algosec.com | `https://www.algosec.com/_api/mcp` | Site Visitor Assistant for site "AlgoSec" v1.0.0 |
| teltonika-networks.com | `https://www.teltonika-networks.com/_api/mcp` | Site Visitor Assistant for site "Teltonika Networks" v1.0.0 |
| turansec.uz | `https://www.turansec.uz/_api/mcp` | Site Visitor Assistant for site "Turan Security \| White Hat" v1.0.0 |
| villageb.io | `https://www.villageb.io/_api/mcp` | Site Visitor Assistant for site "Villageb.io" v1.0.0 |

All four are identical Wix "Site Visitor Assistant" servers (v1.0.0), running at the Wix-standard path `/_api/mcp`.

## Authentication

**None required.** The servers respond to anonymous JSON-RPC `initialize` requests and expose their full tool list without any credentials. The `GenerateVisitorToken` tool creates visitor sessions with no authentication.

## Exposed Tools

Each server exposes 8 tools:

| Tool | Purpose | Risk |
|------|---------|------|
| `SearchInSite` | Full-text search of site content | Low — public content |
| `GetBusinessDetails` | Business name, address, phone, email, hours, schedule | Low — typically public |
| `SearchSiteApiDocs` | Discover available Wix REST APIs for the site | Medium — reveals site capabilities and data model |
| `GenerateVisitorToken` | Create anonymous visitor session token | Medium — enables all API calls below |
| `CallWixSiteAPI` | Call arbitrary Wix REST APIs as a visitor | **Elevated** — can query site data, interact with site features |
| `ReadFullDocsArticle` | Fetch Wix developer documentation | Low |
| `ReadFullDocsMethodSchema` | Fetch API method schemas | Low |
| `BrowseWixRESTDocsMenu` | Browse Wix REST API documentation menu | Low |

## Security Implications

### What an anonymous MCP client can do

1. **Enumerate site features** — `GetBusinessDetails` reveals which Wix features are enabled (e-commerce, bookings, CMS, etc.)
2. **Query site data** — `CallWixSiteAPI` with a visitor token can query any Wix REST API that supports visitor-level access
3. **Search content** — `SearchInSite` provides full-text search across all site content
4. **Discover APIs** — `SearchSiteApiDocs` and `BrowseWixRESTDocsMenu` reveal the full API surface available for the site

### What it cannot do

- No admin-level access — scoped to visitor permissions
- No write operations beyond what a normal site visitor could do (e.g., submitting forms, booking appointments)
- No access to site management, analytics, or admin data

### The core issue

**Site owners likely did not opt into exposing an MCP endpoint.** This appears to be a Wix platform-level feature enabled by default. The `llms.txt` on Wix sites (e.g., turansec.uz) states:

> "This site is powered by Wix and supports the Model Context Protocol (MCP) for agentic AI access."

This text and the MCP endpoint are generated by Wix, not the site owner. Security-focused organizations (these are CVE Numbering Authorities) may not be aware that their website exposes a programmatic interface for AI agents.

### Broader context

Out of 487 CNA domains scanned:
- **0** have intentionally deployed MCP servers
- **4** have MCP servers via Wix (platform-imposed)
- **16** mention MCP in their llms.txt (mostly as installable developer tools, not hosted endpoints)
- Several (neo4j, clickhouse, dremio, yugabyte) offer MCP servers as developer tools but don't host them at their website

The Wix MCP feature is documented at: `https://dev.wix.com/docs/develop-websites/articles/get-started/about-the-wix-site-mcp`

## Methodology

1. Extracted 487 website domains from SecID disclosure registry (CNA entries)
2. Sent MCP JSON-RPC `initialize` requests (POST) to `/{domain}/mcp` and `/{domain}/_api/mcp`
3. Validated responses for proper JSON-RPC format with `serverInfo`
4. Followed up with `tools/list` to enumerate exposed capabilities
5. Scanned saved `llms.txt` / `llms-full.txt` content for MCP and API mentions

## Recommendations

1. **For Wix:** Consider making the MCP endpoint opt-in rather than default, or at minimum make it prominently visible in site settings so owners know it exists
2. **For site owners on Wix:** Review whether the `/_api/mcp` endpoint should be active on your site. Check what data is accessible via visitor-level API calls
3. **For the MCP ecosystem:** This is an early example of platform-level MCP deployment. The pattern of "platform adds MCP without explicit site owner consent" will likely repeat as other website builders adopt MCP
